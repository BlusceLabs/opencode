## Goal

Add a new default theme named **varth** — built from jungle green, lemon chiffon, dark black, and dark brown — to both the TUI and the desktop/web app, and make it the active default across both surfaces.

## Constraints

- No new runtime dependencies; only new JSON theme files + edits to existing theme-registration/default-selection code.
- Follow existing theme file conventions exactly (TUI: `defs`+`theme` like `gruvbox.json`; desktop: `palette`+`overrides` light/dark like `nord.json`).
- Preserve the legacy `oc-1`→`oc-2` normalize migration and keep `oc-2.json` selectable (no wiping of stored user prefs).
- Do not touch TUI `provider.id === "opencode"` provider-identity checks — those are unrelated to theming.
- Adhere to repo style guide (no `any`, functional array methods, `const`, no import aliases, no star imports).
- Tests run from package dirs, never repo root.

## Risks

- **Stored `oc-2` prefs on existing users** — likelihood low (fork, few users), impact minor (they keep oc-2 until they change it). Mitigation: we only change the *unset* default; `normalize` and stored prefs are preserved.
- **Missed `oc-2` reference somewhere** — likelihood medium (10 sites in `context.tsx` alone). Mitigation: post-edit `grep -rn '"oc-2"' packages/ui packages/app packages/desktop` to confirm only the intentional `oc-2.json` file + legacy `normalize` remain.
- **`windows.ts` background-base token name mismatch** — likelihood low. Mitigation: re-read `windows.ts` + `resolve.ts` before editing to confirm the exact resolved-token key.
- **TUI `theme.test.ts` asserts `opencode` default** — likelihood medium. Mitigation: read the test in Chunk 3 and update the assertion to `varth` if it pins the default.
